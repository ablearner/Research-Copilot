# Research-Copilot 系统完整运行流程说明

这份文档解释当前系统一次请求怎样从前端、API、Supervisor Graph、工具层一路跑到结果返回。

如果你只想启动项目，请先看 [系统运行指南.md](系统运行指南.md)。

## 1. 一句话总览

`Research-Copilot` 是一个由高层 research runtime 编排、由底层 `RagRuntime` 执行具体文档与检索动作的科研助手：

```text
前端科研工作区
-> FastAPI routers
-> LiteratureResearchService
-> ResearchSupervisorGraphRuntime
-> specialist agents / research services
-> RagRuntime
-> DocumentTools / RetrievalTools / AnswerTools / ChartTools
-> vector store / graph store / session memory / local persistence
```

系统分成两层：

- 上层：research 任务决策、工作流推进、workspace 与 conversation 管理
- 下层：文档、图表、检索、回答等稳定执行

## 2. 当前主链路

### 2.1 Research Discovery

```text
LiteratureResearchPanel
-> POST /research/agent
-> LiteratureResearchService
-> ResearchSupervisorGraphRuntime.run()
-> supervisor_node
-> create_research_task
-> AutonomousResearchRuntime / LiteratureScoutAgent / PaperSearchService
-> TopicPlanningSkill / ResearchQueryRewriteSkill
-> PaperRankingSkill / PaperCurationSkill
-> SurveyWritingSkill / ReviewWritingSkill
-> candidate papers + report + workspace
```

这一步主要做：

- 主题理解和用户意图解析（`ResearchUserIntentResolverSkill`）
- query planning（`TopicPlanningSkill` / `ResearchQueryRewriteSkill`）
- 外部学术源搜索（arXiv、OpenAlex、Semantic Scholar、IEEE、Zotero）
- 去重、排序、候选论文池生成（`PaperRankingSkill` / `PaperCurationSkill`）
- 研究草稿与报告写回（`SurveyWritingSkill` / `ReviewWritingSkill` / `WritingPolishSkill`）

### 2.2 Research Import

```text
前端勾选论文
-> POST /research/tasks/{task_id}/papers/import/jobs
-> LiteratureResearchService.start_import_job()
-> PaperImportService
-> download PDF
-> RagRuntime.handle_parse_document()
-> RagRuntime.handle_index_document()
-> embedding index + graph index
-> 更新 imported docs / workspace / report
```

这一步做的是：

- 下载 PDF 到本地
- 解析文档
- 建 embedding 索引
- 建图谱索引
- 更新任务状态和已导入文档集合

### 2.3 Research Collection QA

```text
前端提问
-> POST /research/agent
-> LiteratureResearchService.ask_task_collection()
-> ResearchQARoutingSkill (QA route 判定)
-> collection_qa | document_drilldown | chart_drilldown
-> AutonomousResearchCollectionQARuntime / 对应 agent chain
-> 回答 + trace + workspace update
```

当前统一对外入口仍然是 `/research/agent`，但内部会根据问题和视觉锚点做路由：

- `collection_qa`
  由 `AutonomousResearchCollectionQARuntime` 驱动，`ResearchKnowledgeAgent` 收集证据，再交给 `ResearchWriterAgent` 组织答案
- `document_drilldown`
  走底层文档问答链路
- `chart_drilldown`
  走图表分析或 fused ask 链路

集合级问答的底层证据融合策略是：

- sparse retrieval
- vector retrieval
- graph retrieval
- graph summary retrieval
- reranker 重排

### 2.4 Advanced Research Actions

```text
前端 compare / recommend / compress
-> POST /research/agent
-> ResearchSupervisorGraphRuntime
-> ResearchFunctionService / PaperAnalysisAgent / PaperAnalysisSkill
-> workspace.metadata updates
-> conversation messages
```

高级动作不会替代 discovery / import / QA 主链路，而是在已有论文池和 workspace 上补充结构化能力：

- 多论文对比
- 优先阅读推荐
- 上下文压缩
- 代码关联（`CodeLinkingSkill`）

当选中的论文已经导入并有 `document_id` 时，分析链路会优先补取 retrieval / graph summary 证据，再交给 `PaperAnalysisAgent` / `PaperAnalysisSkill`。

## 3. FastAPI 启动流程

入口在 [apps/api/main.py](apps/api/main.py)。

启动顺序大致是：

1. 读取 `Settings`
2. 构建 `RagRuntime`（`build_graph_runtime()` -> `build_rag_runtime()`）
3. 注册 MCP 外部工具（`academic-search`、可选 `zotero-local`）
4. 构建 `LiteratureResearchService`（通过 [apps/api/research_runtime.py](apps/api/research_runtime.py)）
5. 注册 `ResearchFunctionService` 并把 research function handlers 和 runtime tool specs 挂载到 `tool_registry`
6. 创建 `MCPServerApp`（对外暴露 RAG 能力）
7. 注册 routers
8. 进入 `lifespan`
9. 按配置初始化底层 store

其中底层装配逻辑主要在 [apps/api/runtime.py](apps/api/runtime.py)：

- `build_rag_runtime()` 装配全套底层依赖
- `initialize_rag_runtime()` 在 lifespan 阶段执行
- vector store 如果有 `ensure_schema()` 会执行
- graph store 如果有 `connect()` 会执行

research 层装配逻辑在 [apps/api/research_runtime.py](apps/api/research_runtime.py)：

- `build_literature_research_service()` 装配学术搜索工具、报告服务、导入服务和多类 research skills
- `register_research_runtime_extensions()` 注册 `ResearchFunctionService` 和 runtime tool specs

这意味着：

- 如果配置了真实 Milvus，但服务没起来，启动可能失败
- 如果配置了 Neo4j，但连接不可用，启动可能失败
- 如果使用 memory graph store / memory session memory，则不会依赖外部服务

## 4. Runtime 装配关系

`apps/api/runtime.py` 会把整套底层依赖装起来：

- LLM adapter（DashScope / OpenAI-compatible / Gemini / Local）
- chart vision adapter（可独立配置）
- embedding adapter（DashScope / OpenAI-compatible / Local hash）
- vector store
- graph store
- reranker（SentenceTransformers CrossEncoder / HeuristicFallback）
- `GraphSessionMemory`
- tool registry + tool executor
- skill registry（从 `skills/specs/` 加载 YAML + `build_core_research_skill_profiles()`）
- prompt resolver（从 `prompts/skill_prompt_mapping.yaml` 加载）
- reasoning strategies（CoT / PlanAndSolve / ReAct）
- MCP client registry（外部工具注册）
- `RagRuntime`

`apps/api/research_runtime.py` 装配 research 层依赖：

- 学术搜索工具（ArXiv / OpenAlex / SemanticScholar / IEEE / Zotero）
- `PaperSearchService` / `PaperImportService` / `ResearchReportService`
- research skills（`PaperReadingSkill` / `SurveyWritingSkill` / `ReviewWritingSkill` / `WritingPolishSkill` / `ResearchEvaluationSkill` / `CodeLinkingSkill`）
- `AutonomousResearchRuntime` / `AutonomousResearchCollectionQARuntime`
- `LiteratureResearchService`
- `ResearchFunctionService`

当前实际支持的底层存储选择：

- vector store
  - `milvus` / `zilliz`
  - `pgvector`
  - `memory`
- graph store
  - `neo4j`
  - `memory`
- runtime session memory
  - `mysql`
  - `memory`
  - `auto`（有 MySQL 配置时用 MySQL，否则退回内存）

## 5. 记忆系统在流程里的位置

项目里同时有 research 侧 memory 和底层 runtime memory。

### 5.1 Research 侧 memory

定义在 `memory/`：

- `WorkingMemory`
  当前会话短期工作状态
- `SessionMemory`
  research 会话级持久化上下文
- `LongTermMemory`
  跨会话长期记忆（默认 JSON 文件型，也支持 Qdrant）
- `PaperKnowledgeMemory`
  单篇论文知识记录
- `UserProfileMemory`
  用户研究偏好，构建在 `LongTermMemory` 之上，由 `PreferenceMemoryAgent` 驱动学习

这一套主要由 [memory/memory_manager.py](memory/memory_manager.py) 统一管理，并在 [sdk/client.py](sdk/client.py) 中装配。

### 5.2 RAG runtime session memory

定义在 [rag_runtime/memory.py](rag_runtime/memory.py)：

- `GraphSessionMemory`
  保存当前文档、最近检索摘要、最近回答摘要、任务意图、图表问答历史等结构化会话快照

这一套主要服务于底层 prompt context 和会话级检索问答连续性。

## 6. CLI 在整体系统里的位置

CLI 入口是 [apps/cli.py](apps/cli.py)，但它不是简单 `curl` 包装。

调用链大致是：

```text
apps/cli.py
-> sdk/client.py
-> LiteratureResearchService
-> MemoryManager / report service / search service / import service
```

CLI 直接复用本地 SDK 和 research service，因此：

- 适合本机开发与调试
- 不依赖前端页面
- 也不依赖 HTTP API 才能运行

## 7. 你最该记住的结论

1. `ResearchSupervisorGraphRuntime` 负责“下一步做什么”。
2. `RagRuntime` 负责“把解析、索引、检索、回答这些动作稳定执行完”。
3. `/research/agent` 是统一主入口，但内部仍然会分发到 discovery、import、collection QA、document drilldown、chart drilldown 等不同链路。
4. 项目里的 memory 不是单一聊天历史，而是分层的 research memory 加上底层 runtime session memory。
5. `skills/research/` 下的 16+ 研究技能类是业务逻辑的关键组件，包括 query planning、paper ranking、survey writing、QA routing、visual anchor 等。
6. MCP 协议同时用于对外接入外部工具（如 Zotero）和对外暴露 RAG 能力（`MCPServerApp`）。
