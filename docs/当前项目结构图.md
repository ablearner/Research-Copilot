# Research-Copilot 当前项目结构图

这份文档只做一件事：把 **当前仓库的真实结构** 讲清楚。

如果你刚接手这个项目，建议把它和下面两份文档配合看：

- [README.md](../README.md)
- [系统完整运行流程说明.md](../系统完整运行流程说明.md)

## 1. 一句话结构

当前项目可以先按这条主线理解：

```text
前端工作区
-> FastAPI API
-> services/research 高层 Supervisor Graph
-> rag_runtime 底层 tool-first RAG runtime
-> tools / retrieval / adapters
-> vector store / graph store / memory / 本地文件 / research 持久化
```

也就是说：

- 高层负责“下一步做什么”
- 底层负责“这一步怎么稳定做完”

## 2. 目录总览

建议先看下面这棵树：

```text
Research-Copilot/
├── app/                       # Next.js 页面入口
├── components/                # 前端研究工作区组件
├── lib/                       # 前端 controller、API 封装、types
├── apps/api/                  # FastAPI 启动、依赖装配、routers
├── agents/                    # Supervisor 和 specialist agents（9 个）
├── services/                  # 顶层服务命名空间（根目录仅保留包入口）
│   └── research/              # 高层 research 业务与编排
├── rag_runtime/               # 底层 RAG runtime（tool-first pipeline）
│   └── services/              # 底层索引、PDF/OCR、图摘要、图社区等支撑服务
├── tools/                     # 文档/图表/检索/回答等稳定工具层
│   └── research/              # 外部学术源搜索工具（ArXiv/OpenAlex/SemanticScholar/IEEE）
├── retrieval/                 # hybrid retrieval / fusion / rerank / evidence
├── adapters/                  # LLM / embedding / vector store / graph store 适配
├── chains/                    # 回答链、图表理解链、图抽取链等链式封装
├── planners/                  # 轻量规划器（router / retrieval / validation）
├── reasoning/                 # reasoning agents（cot / plan-and-solve / react）
├── skills/                    # skill spec、skill registry、research skill 扩展
│   └── research/              # 16+ 研究技能类（query planning / paper ranking / survey writing 等）
├── tooling/                   # 工具注册 / 执行框架、research function 规范
├── mcp/                       # MCP 客户端注册和服务端暴露
├── sdk/                       # 本地 CLI / SDK 调用封装和 runtime profile
├── domain/schemas/            # Pydantic schema
├── memory/                    # 多层 memory: working/session/long-term/user-profile/paper-knowledge
├── repositories/              # repository 抽象（当前大部分为占位 stub）
├── prompts/                   # prompts 模板
├── scripts/                   # 环境检查、Milvus 重置、SciGraphQA 入库等工具脚本
├── tests/                     # 单测与集成测试
├── evaluation/                # 评测框架：runner/metrics/benchmarks/datasets
├── .windsurf/                 # Windsurf workflows
└── .data/                     # 本地持久化数据、上传文件、Milvus 数据等
```

这是一个比较轻量的辅助层。

### 3.7 `reasoning/`

这层不是空壳。

它现在承载的是几类还在实际使用的 reasoning agent：

- `CoTReasoningAgent`
- `PlanAndSolveReasoningAgent`
- `ReActReasoningAgent`

当前使用方式大致是：

- `AnswerTools` 使用 `CoTReasoningAgent`
- `LiteratureScoutAgent` 和 `ResearchKnowledgeAgent` 使用 `PlanAndSolveReasoningAgent`

所以 `reasoning/` 的定位更接近：

**可复用 reasoning 策略层**

### 3.9 `memory/`

这层负责研究侧的多层记忆管理：

- [memory/memory_manager.py](../memory/memory_manager.py) — 统一入口，协调各层 memory
- [memory/working_memory.py](../memory/working_memory.py) — 当前会话内工作记忆
- [memory/session_memory.py](../memory/session_memory.py) — 会话级记忆（JSON / 内存）
- [memory/long_term_memory.py](../memory/long_term_memory.py) — 长期记忆（JSON / 内存 / Qdrant）
- [memory/user_profile_memory.py](../memory/user_profile_memory.py) — 用户研究偏好画像
- [memory/paper_knowledge_memory.py](../memory/paper_knowledge_memory.py) — 论文知识记忆

当前 memory 架构是分层的：

- **Working Memory**：当前 task scope 内的临时状态
- **Session Memory**：跨 turn 的会话上下文
- **Long-term Memory**：跨会话的研究知识和结论
- **User Profile**：用户研究兴趣、偏好作者/期刊/会议、检索策略
- **Paper Knowledge**：论文级别的知识提取结果

### 3.10 `tooling/`

工具注册 / 执行框架和 research function 规范。

它负责：

- `ToolSpec`、`ToolRegistry`、`ToolExecutor` 的定义和管理

- Supervisor 侧 tool specs 生成
- `research_supervisor_tool_specs.py` — 高层 action 的 ToolSpec 包装

### 3.11 `mcp/`

- 客户端：`MCPClientRegistry` 注册外部工具（当前主要是 `academic-search` 和 `zotero-local`）

### 3.12 `sdk/`

## 4. 两条主线怎么跑

### 4.1 高层主线

```text
前端
-> /research/agent
-> LiteratureResearchService
-> ResearchSupervisorGraphRuntime
-> supervisor_node
-> specialist agent / research service
```

这条线负责：

- 搜论文
- 生成报告
- 发起导入 job
- collection QA
- 对比、推荐、上下文压缩

补一条当前代码里的真实入口约束：

- 前端不会直接请求 FastAPI，而是先打到 `app/api/backend/[...path]/route.ts`
- 这个代理再转发到 `FASTAPI_BASE_URL`，默认是 `http://127.0.0.1:8000`
- 因此前端调试要同时关注 Next.js 终端和 FastAPI 终端

---

### 4.2 底层主线

```text
research service / API
-> RagRuntime
-> DocumentTools / RetrievalTools / AnswerTools
-> retrieval / adapters / stores
```

这条线负责：

- PDF 解析
- chunk 切分
- embedding 建库
- graph extraction / graph index
- hybrid retrieval
- evidence bundle
- grounded answer

## 5. 现在最容易混淆的地方

### 5.1 `services/research/` 和 `rag_runtime/`

区别是：

- `services/research/`：决定研究任务怎么推进
- `rag_runtime/`：把文档/图表/RAG 动作执行完

一句话：

**前者是业务编排，后者是底层执行。**

### 5.2 `tools/` 和 `retrieval/`

区别是：

- `tools/`：给 runtime 调的稳定工具
- `retrieval/`：工具内部实际用到的检索实现

### 5.3 `agents/` 和 `services/research/`

区别是：

- `agents/`：Supervisor 和 specialist 的角色逻辑
- `services/research/`：任务状态、持久化、job、流程收口

### 5.4 `planners/`、`reasoning/`、`skills/`

这三层不要混成一类：

- `planners/`：轻量 planner，只负责 router / retrieval / validation 这类结构化规划
- `reasoning/`：真正的 reasoning 策略实现，直接和 LLM 行为相关
- `skills/`：任务能力配置和扩展层，决定 prompt、retrieval policy、memory policy 等

一句话：

**`planners/` 偏“流程内规划”，`reasoning/` 偏“推理方式”，`skills/` 偏“能力配置”。**

## 7. 启动与运行边界

如果你不是只读代码，而是要真的把项目跑起来，建议先记住这几个事实：

- FastAPI 启动入口是 [apps/api/main.py](../apps/api/main.py)
- 开发脚本是 [scripts/run_api_dev.sh](../scripts/run_api_dev.sh)
- CLI 入口是 [apps/cli.py](../apps/cli.py)
- 配置入口是 [core/config.py](../core/config.py)

当前最重要的运行边界：

- `.env` 默认在项目根目录读取
- `APP_ENV=local` 时才会暴露 `/uploads/*`
- `VECTOR_STORE_PROVIDER` 默认是 `milvus`
- `GRAPH_STORE_PROVIDER` 默认是 `memory`
- `SESSION_MEMORY_PROVIDER=auto` 时，有 MySQL 配置就接 MySQL，没有就退回内存
- FastAPI `lifespan` 启动阶段会初始化 graph runtime；如果你配置了真实外部 store，但依赖不可用，启动会直接失败

推荐最小启动顺序：

1. 先跑 `python scripts/check_env.py`
2. 再跑 `bash scripts/run_api_dev.sh`
3. 最后跑 `npm run dev`

## 8. 一句话总结

当前项目的结构已经可以稳定地理解成：

**`services/research/` 负责高层研究编排，`rag_runtime/` 负责底层 tool-first RAG 执行，`tools/ + retrieval/ + adapters/` 负责具体能力与基础设施，`skills/research/` 提供可复用的业务技能单元，`memory/` 提供多层记忆管理，`mcp/` 负责外部工具集成和能力输出，`evaluation/` 提供研究质量评测框架。**
