# Kepler 当前项目结构图

这份文档只做一件事：把 **当前仓库的真实结构** 讲清楚。

如果你刚接手这个项目，建议把它和下面两份文档配合看：

- [README.md](../README.md)
- [系统完整运行流程说明.md](../系统完整运行流程说明.md)

## 1. 一句话结构

当前项目可以先按这条主线理解：

```text
前端工作区
-> FastAPI API
-> services/research 高层 Supervisor Graph
-> rag_runtime 底层 tool-first RAG runtime
-> tools / retrieval / adapters
-> vector store / graph store / memory / 本地文件 / research 持久化
```

也就是说：

- 高层负责“下一步做什么”
- 底层负责“这一步怎么稳定做完”

## 3. 最重要的几个目录

### 3.1 `web/` (当前活跃前端)

这是当前实际使用的前端，基于 **Next.js 15 (App Router) + React 18 + TypeScript + Tailwind CSS** 构建。

目录结构：

```text
web/
├── app/
│   ├── layout.tsx             # Next.js 根布局
│   ├── page.tsx               # 首页（加载 App 组件）
│   ├── globals.css            # 全局样式
│   └── api/files/[...path]/
│       └── route.ts           # 图片文件 Route Handler
├── next.config.ts             # Next.js 配置（含 API rewrites）
├── tailwind.config.js         # Tailwind 主题配置
├── package.json               # 前端依赖
└── src/
    ├── App.tsx                # 主应用：会话管理、侧边栏、路由
    ├── api.ts                 # 后端 API 封装（sendMessage / 会话 CRUD）
    ├── types.ts               # TypeScript 类型定义
    └── components/
        ├── ChatView.tsx       # 聊天视图：消息过滤、发送、task_id 跟踪
        ├── MessageBubble.tsx  # 消息气泡：Markdown 渲染、内联图片、论文卡片、决策轨迹、数学公式 / 代码高亮
        ├── InputBar.tsx       # 输入栏（含文件上传触发器）
        ├── PaperCard.tsx      # 论文卡片组件（支持勾选导入）
        ├── FileUpload.tsx     # 文件上传组件（拖拽 + 点击）
        ├── Toast.tsx          # 轻量级通知组件
        ├── Sidebar.tsx        # 侧边栏：会话列表
        └── WelcomeScreen.tsx  # 欢迎页
```

关键特性：

- **内联图片显示**：自动检测消息中的 `.data/storage/...` 路径，通过 Next.js Route Handler 转为 `/api/files/` URL 并渲染为 `<img>` 标签
- **消息过滤**：区分用户可见内容（answer/report/论文分析结果）和内部通知（Manager/Agent 决策轨迹）
- **task_id 跟踪**：前端维护 `taskId` 状态，后续消息自动携带，支持图表分析等上下文相关操作
- **数学公式渲染**：集成 KaTeX，支持 `$...$` 行内和 `$$...$$` 块级 LaTeX 公式
- **代码高亮**：集成 react-syntax-highlighter，支持多语言代码块渲染
- **文件上传**：拖拽 + 点击上传 PDF 文件
- **API 代理**：通过 Next.js rewrites 代理请求到 FastAPI `http://127.0.0.1:8000`

### 3.7 `reasoning/`

### 3.9 `memory/`

这层负责研究侧的多层记忆管理：

- [memory/memory_manager.py](../memory/memory_manager.py) — 统一入口，协调各层 memory
- [memory/working_memory.py](../memory/working_memory.py) — 当前会话内工作记忆
- [memory/session_memory.py](../memory/session_memory.py) — 会话级记忆（JSON / 内存）
- [memory/long_term_memory.py](../memory/long_term_memory.py) — 长期记忆（JSON / 内存 / Qdrant）
- [memory/user_profile_memory.py](../memory/user_profile_memory.py) — 用户研究偏好画像
- [memory/paper_knowledge_memory.py](../memory/paper_knowledge_memory.py) — 论文知识记忆
- [memory/security.py](../memory/security.py) — 记忆内容安全扫描（prompt 注入检测、不可见 unicode 字符检测）

当前 memory 架构是分层的：

- **Working Memory**：当前 task scope 内的临时状态
- **Session Memory**：跨 turn 的会话上下文
- **Long-term Memory**：跨会话的研究知识和结论（upsert 前自动安全扫描）
- **User Profile**：用户研究兴趣、偏好作者/期刊/会议、检索策略
- **Paper Knowledge**：论文级别的知识提取结果
- **Frozen Snapshot**：会话首次 hydrate 时冻结的不可变记忆快照，用于稳定 system prompt prefix（支持 Anthropic prompt caching）

### 3.11 `mcp/`

MCP（Model Context Protocol）客户端注册和服务端暴露。

- 客户端：`MCPClientRegistry` 注册外部工具（当前主要是 `academic-search` 和 `zotero-local`）
- 服务端：`MCPServerApp` 把 RAG 能力暴露给外部系统

**Hermes 式改进（新增模块）**：

- [mcp/client/stdio_client.py](../mcp/client/stdio_client.py) — Stdio 传输 MCP 客户端，支持外部进程连接、自动重连（指数退避）、超时控制
- [mcp/client/http_client.py](../mcp/client/http_client.py) — HTTP/SSE 传输 MCP 客户端，支持 Streamable HTTP 协议
- [mcp/security.py](../mcp/security.py) — MCP 安全层：安全环境变量过滤（subprocess 只传白名单 key）、错误消息凭据脱敏、tool description 注入检测
- `MCPClientRegistry.register_from_config()` — 从配置字典批量创建 stdio/http 客户端
- `MCPClientRegistry.scan_all_tools()` — 扫描所有外部 tool description 检测注入模式

### 3.15 `context/`（Hermes 式新增）

上下文管理引擎，借鉴 Hermes 的 context compression + token budgeting 架构。

- [context/compressor.py](../context/compressor.py) — 上下文压缩引擎：旧 tool 结果裁剪 → 中间轮 LLM 摘要 → tool_call/result 对清洗。内置 anti-thrashing 守卫（压缩间隔 ≥ 3 轮 + cooldown 30s）
- [context/token_counter.py](../context/token_counter.py) — Token 预算管理：模型 context length 查表（GPT-4 turbo 128k 等）、消耗跟踪、overflow probing（遇 413 自动缩减预算 15%）
- [context/prompt_caching.py](../context/prompt_caching.py) — Anthropic prompt caching：在 system + 前 N 轮消息插入 `cache_control` 断点，最大化 prefix hit

### 3.16 `security/`（Hermes 式新增）

安全层，防止敏感数据泄露到日志、记忆或 LLM 压缩上下文中。

- [security/redact.py](../security/redact.py) — 敏感数据脱敏：正则检测 API key、Bearer token、密码、数据库连接串、环境变量机密，替换为 `[REDACTED_*]` 占位符

### 4.1.1 用户请求的完整决策链

一条用户消息从进入到输出的完整处理过程：

```text
用户消息
│
├─ Step 1: 意图解析 (ResearchUserIntentResolverSkill)
│   ├─ 启发式扫描: 关键词 marker 匹配 → intent + confidence
│   ├─ confidence ≥ 0.8 → 直接返回（跳过 LLM）
│   └─ confidence < 0.8 → LLM generate_structured 精化
│
├─ Step 2: 构建 SupervisorState（40+ 字段）
│   ├─ user_intent, route_mode, topic_continuity_score
│   ├─ has_task, candidate_papers, imported_document_count
│   └─ active_thread_id, new_topic_detected, ...
│
├─ Step 3: Supervisor 决策 (_decide_with_llm)
│   ├─ [1] _intent_guardrail_decision()     ← 启发式快路径，0 LLM
│   │   匹配: literature_search / general_answer / paper_import / ...
│   ├─ [2] _guardrail_decision()            ← 状态驱动快路径，0 LLM
│   │   匹配: 文档未解析 / 图表未理解 / 上下文超限 / ...
│   ├─ [3] _context_exceeds_budget()        ← 强制 compress_context
│   └─ [4] LLM generate_structured()        ← 完整 LLM 决策
│       输入: state_snapshot + available_actions + recent_results
│       输出: ResearchSupervisorLLMDecision(action, thought, rationale)
│
├─ Step 4: 路由到 Specialist Agent
│   supervisor_node → specialist_node（如 literature_scout_node）
│
├─ Step 5: Specialist 执行
│   各 specialist 内部可能调用: query planning / ranking / LLM 生成
│
└─ Step 6: 结束判定 (_should_force_finalize)
    ├─ terminal task (general_answer/answer_question/analyze_papers/...) → 直接 finalize
    ├─ search + report 完成 → finalize
    └─ 其他 → 回 supervisor_node 再决策
```

**关键设计**：guardrail 是启发式快路径，只覆盖高置信度场景；低置信度、复合意图、需要推理的路由选择 100% 由 LLM supervisor 决策。

## 5. 现在最容易混淆的地方

### 5.1 `services/research/` 和 `rag_runtime/`

区别是：

- `services/research/`：决定研究任务怎么推进
- `rag_runtime/`：把文档/图表/RAG 动作执行完

一句话：

**前者是业务编排，后者是底层执行。**

### 5.2 `tools/` 和 `retrieval/`

区别是：

- `tools/`：给 runtime 调的稳定工具
- `retrieval/`：工具内部实际用到的检索实现

### 5.3 `agents/` 和 `services/research/`

区别是：

- `agents/`：Supervisor 和 specialist 的角色逻辑
- `services/research/`：任务状态、持久化、job、流程收口

### 5.4 `planners/`、`reasoning/`、`skills/`

这三层不要混成一类：

- `planners/`：轻量 planner，只负责 router / retrieval / validation 这类结构化规划
- `reasoning/`：真正的 reasoning 策略实现，直接和 LLM 行为相关
- `skills/`：任务能力配置和扩展层，决定 prompt、retrieval policy、memory policy 等

一句话：

**`planners/` 偏“流程内规划”，`reasoning/` 偏“推理方式”，`skills/` 偏“能力配置”。**

## 7. 启动与运行边界

如果你不是只读代码，而是要真的把项目跑起来，建议先记住这几个事实：

- FastAPI 启动入口是 [apps/api/main.py](../apps/api/main.py)
- 开发脚本是 [scripts/run_api_dev.sh](../scripts/run_api_dev.sh)
- CLI 入口是 [apps/cli.py](../apps/cli.py)
- 配置入口是 [core/config.py](../core/config.py)
- Zotero Bridge 脚本是 [scripts/wsl_zotero_bridge.sh](../scripts/wsl_zotero_bridge.sh)

当前最重要的运行边界：

- `.env` 默认在项目根目录读取
- `APP_ENV=local` 时才会暴露 `/uploads/*`
- `VECTOR_STORE_PROVIDER` 默认是 `milvus`
- `GRAPH_STORE_PROVIDER` 默认是 `neo4j`
- `SESSION_MEMORY_PROVIDER=auto` 时，有 MySQL 配置就接 MySQL，没有就退回内存
- `STORAGE_PROVIDER` 默认 `json`，可切换为 `sqlite`（SQLite WAL 模式）
- `CORS_ALLOW_ORIGINS` 默认允许 `localhost:3000/3001`
- `RATE_LIMIT_MAX_REQUESTS` 默认 60 次/分钟/IP
- `JSON_LOG_FORMAT=true` 启用结构化 JSON 日志（默认关闭）
- FastAPI `lifespan` 启动阶段会初始化 vector store 和 graph store；如果依赖不可用（如 Milvus/Neo4j 未启动），后端会卡在 `Waiting for application startup`

### 7.2 WSL 环境注意事项

- WSL 默认只分配宿主机一半内存，建议在 `C:\Users\<用户名>\.wslconfig` 中配置：

## 10. 性能优化

针对 Supervisor Graph 多 LLM 调用导致的延迟问题，已实施以下优化：

### 10.1 已完成优化

| 优化 | 位置 | 效果 |
|------|------|------|

| 场景 | 优化前 | 优化后 |
|------|--------|--------|
| "你好"（闲聊） | 4 次 LLM | 2 次 |
| "调研 LLM agent 论文" | 3-4 次 | 1-2 次 |
| "这篇论文讲了什么" | 3-4 次 | 1-2 次 |

### 10.2 候选优化方案

| 优先级 | 方案 | 原理 | 预期收益 |
|--------|------|------|----------|

## 11. Kepler vs Hermes 架构对比

| 维度 | Kepler | Hermes |
|------|--------|--------|
| 意图识别 | 独立模块，启发式 + LLM 两阶段 | 无独立模块，LLM 内隐完成 |
| 路由决策 | 3 层：guardrail → state check → LLM supervisor | LLM 自选 tool（tool_choice="auto"） |
| 查询规划 | 专门 Skill：topic normalization → LLM rewrite → 同义词扩展 | 无，LLM 直接生成搜索参数 |
| 执行模式 | Supervisor Graph：决策 → 派发 → 执行 → 回收 → 再决策 | 扁平循环：LLM → tool → LLM → tool → ... |
| 状态管理 | 40+ 字段结构化状态 + 论文/文档/图表追踪 | 纯对话历史（messages 数组） |
| 结束条件 | `_should_force_finalize()` 多条件判定 | 无 tool_call 即停 |
| 最少 LLM 调用 | 2 次（guardrail 路由 + specialist 执行） | 1 次（直接选 tool 或直接回答） |
| 最大优势 | 可控性强、可追踪、领域精确 | 响应快、架构简单 |
| 最大劣势 | 每层都有延迟开销 | 无法精细控制路由，依赖模型能力 |

**一句话总结**：Kepler 用确定性分层换可控性，Hermes 用模型能力换速度。
