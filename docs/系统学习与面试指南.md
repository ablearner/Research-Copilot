# Research-Copilot 系统学习与面试指南

这份文档用于把当前项目讲清楚，而不是教你怎么启动它。

## 1. 一句话介绍

`Research-Copilot` 是一个多智能体科研助手：上层由 `ResearchSupervisorGraphRuntime` 的 supervisor orchestration 统一做研究任务决策，下层复用文档解析、图表理解和 GraphRAG 工具底座，完成论文发现、导入、研究集合问答、高级对比推荐和 TODO 闭环。

## 2. 项目价值怎么讲

这个项目的价值不在“能问答”，而在下面四点：

- 不是单 prompt，而是 `manager + runtime + tool layer`
- 不是单纯向量检索，而是 `sparse + vector + graph + graph summary`
- 不是单次回答，而是 `task + report + workspace + job + conversation`
- 不是单后端，而是有可持续使用、可恢复回放的研究工作区

## 3. 当前最重要的链路

### 3.1 研究发现

```text
用户问题
-> /research/agent
-> supervisor_node
-> LiteratureScoutAgent / PaperSearchService
-> candidate papers + report
```

### 3.2 论文导入

```text
用户勾选论文
-> /research/tasks/{task_id}/papers/import/jobs
-> download PDF
-> parse
-> index
-> graph
-> update task/report
```

### 3.3 研究集合问答

```text
用户提问
-> /research/agent
-> supervisor_node 决策 QA 路径
-> ResearchKnowledgeAgent / ResearchWriterAgent / 底层 drilldown runtime
-> retrieve collection evidence [if imported docs exist]
-> build collection manifest [papers + report]
-> answer collection question
-> grounded answer + todo update
```

### 3.4 高级对比与推荐

```text
用户点击高级动作
-> /research/agent (advanced_action=compare/recommend)
-> compare_papers / recommend_papers / compress_context
-> workspace metadata + conversation messages
-> 刷新后研究线程回放
```

## 4. 关键架构判断

### 4.1 为什么只保留一个高层决策中心

当前系统明确让 `supervisor_node` 负责“下一步做什么”，这样：

- 不会出现多个决策中心互相抢控制权
- trace 更稳定
- workspace 更容易收敛
- 出错时更容易定位

### 4.2 为什么还保留 RagRuntime

因为它承担的是稳定执行：

- parse
- index
- retrieval
- grounded answer
- chart understanding

也就是“怎么稳定做完”，而不是“做什么”。

### 4.3 为什么 collection QA 不重新外部搜论文

因为这一步是在当前研究任务内部做证据整合，而不是重新开一轮 discovery。

边界非常清楚：

- 外部搜索解决“找什么论文”
- collection QA 解决“当前任务里的文档、论文池和报告，能支持什么结论”

## 5. 面试时的主技术点

### 5.1 多智能体不是堆数量

这里的重点不是 agent 数量（8 个），而是职责边界：

- `ResearchSupervisorAgent` 决策与委派
- `LiteratureScoutAgent` 执行发现与搜索扩展
- `ResearchKnowledgeAgent` 执行证据组织、上下文压缩与集合检索
- `ResearchWriterAgent` 执行综述、回答和 TODO 生成
- `PaperAnalysisAgent` 执行对比、推荐和 explanation
- `ChartAnalysisAgent` 执行 figure 列表、visual anchor 和单图分析
- `PreferenceMemoryAgent` 负责用户偏好学习与记忆，驱动 `UserProfileMemory` 更新
- `GeneralAnswerAgent` 负责通用问答
- `compare_papers` / `recommend_papers` 作为高级研究动作经 `ResearchFunctionService` 注册执行

### 5.2 RAG 不是只靠向量

当前系统的 RAG 是混合检索：

- 词法检索：BM25 风格 sparse retrieval
- 向量检索：Milvus
- 图谱检索：内存图存储或 Neo4j
- 图摘要检索：Graph summary

这些结果会先按证据对象合并，再用 `Cross-Encoder` 做统一重排（不可用时退回 `HeuristicFallbackReranker`），最后进入 `EvidenceBundle`。

### 5.4 不是只有 agents，还有 skills

`skills/research/` 下有 16+ 研究技能类，是 agents 和 services 的业务逻辑基座：

- `TopicPlanningSkill` / `ResearchQueryRewriteSkill`：查询规划与改写
- `PaperRankingSkill` / `PaperCurationSkill`：论文排序与筛选
- `SurveyWritingSkill` / `ReviewWritingSkill` / `WritingPolishSkill`：综述写作与润色
- `ResearchQARoutingSkill`：QA 路由判定
- `PaperAnalysisSkill` / `PaperReadingSkill`：论文分析与阅读
- `ResearchVisualAnchorSkill`：视觉锚点识别
- `ResearchUserIntentResolverSkill`：用户意图解析
- `CodeLinkingSkill`：代码关联
- `ResearchEvaluationSkill`：研究质量评估

这一层的定位是：可复用的业务逻辑单元，agents 和 services 负责编排，skills 负责执行。

### 5.5 MCP 协议同时用于对内和对外

- 对内：MCP 客户端注册外部工具（当前主要是 `academic-search` 和 `zotero-local`）
- 对外：`MCPServerApp` 把 RAG 能力暴露给外部系统

本地能力直调，外部系统才走 MCP。

### 5.3 状态不是散在各处

当前系统把 `workspace` 提升成第一等状态，显式维护：

- `current_stage`
- `status_summary`
- `key_findings`
- `evidence_gaps`
- `next_actions`
- `stop_reason`

这让系统更像任务系统，而不是聊天壳。

同时，`conversation snapshot + conversation messages` 负责前端恢复：

- snapshot 保存当前 workspace、advanced strategy、task/import/ask 结果
- messages 保存候选论文、报告、回答、对比、推荐、压缩摘要和 trace
- 前端刷新后优先回放 messages，避免只靠快照拼页面

## 6. 简历里可以怎么写

一种简洁写法：

> 设计并实现 `Research-Copilot`，构建以 Supervisor Graph 为单一高层决策中心的8个专家 Agent + 16+ 研究技能多智能体科研助手；打通论文发现、开放 PDF 导入、文档/图表 GraphRAG 检索、研究集合 grounded QA 和高级对比推荐；通过 workspace、trace、job、conversation 持久化和前端消息回放实现可追踪的研究闭环；通过 MCP 协议对外接入 Zotero 等外部工具并暴露 RAG 能力。

如果要更工程化：

> 基于 FastAPI、LangGraph、Milvus、Neo4j 和 Next.js 实现多智能体科研助手；完成 research discovery、paper import、collection QA、paper comparison、reading recommendation 等链路，并将文档解析、图表理解和混合检索收敛为稳定工具层；通过 MCP 协议支持外部工具集成和能力输出。

## 7. 追问时怎么答

### 7.1 为什么不是把底座也做成一个大 agent

因为 parse/index/retrieve 这类流程更适合稳定执行图，但当前执行图中的节点衔接也已改为 LLM 基于状态选择下一跳，而不是外部固定编排。

### 7.2 导入为什么慢

因为导入包含：

- PDF 下载
- 解析
- 图谱抽取
- embedding
- 向量和图写入

当前已经通过小并发和更大的 embedding batch 做过优化。

### 7.3 为什么回答有时会提示证据不足

因为 collection QA 不会重新去外部搜论文，核心证据仍来自当前任务已有内容。问题太宽、导入太少、候选论文和报告里的间接证据又不够直接时，系统会显式暴露 gap，而不是无依据强答。
