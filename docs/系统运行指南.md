## 1. 当前系统由哪些部分组成

`Research-Copilot`不是单一后端服务，而是由几层组成的本地联调系统：

- 前端：`Next.js 15 (App Router) + React 18 + TypeScript + Tailwind CSS`（`web/` 目录）
- API：`FastAPI`
- 高层编排：`services/research/` 下的 Supervisor Graph 风格 research runtime
- 底层执行：`rag_runtime/` 下的 tool-first RAG runtime
- 向量库：默认 `Milvus`
- 图存储：默认 `Neo4j`，也支持内存
- 会话记忆：默认 `auto`，即内存或 `MySQL`

前后端的本地默认端口：

- 前端：`http://127.0.0.1:3000`
- 后端：`http://127.0.0.1:8000`
- Milvus：`http://127.0.0.1:19530`
- Neo4j HTTP：`http://127.0.0.1:7474`
- Neo4j Bolt：`bolt://127.0.0.1:7687`
- Zotero Bridge：`http://127.0.0.1:23119`（WSL 环境通过 socat 转发）
- MySQL：`127.0.0.1:3306`（仅在启用 MySQL session memory 时需要）

当前前端（`web/`）通过 Next.js rewrites 代理请求到 FastAPI：

- `web/src/api.ts` 请求 `/api/backend/*`，由 `next.config.ts` 转发到 `http://127.0.0.1:8000`
- 图片文件通过 Next.js Route Handler (`web/app/api/files/[...path]/route.ts`) 从 `.data/storage/` 读取并以 `/api/files/` 路径提供

只启动前端、不启动后端时，前端 API 请求会失败。

## 2.1 当前工具分层原则

当前项目已经按下面这条原则收敛：

- 本地能力直调
- 外部系统才走 `MCP`

具体来说：

- 本地 runtime tools
  - `academic_search`
  - `local_file`
  - `web_search`
  - `notification`
  - `library_sync`
  - `search_or_import_paper`
  - `code_execution`

  这些能力都直接注册到本地 `tool_registry`，本质是当前进程内的 Python handler，不依赖外部 MCP server。

- 外部 MCP tools
  - 当前实际接入的外部能力主要是 `zotero-local`

  这类能力通过 `MCPClientRegistry` 接入，适合跨进程、跨应用、外部系统桥接场景。

这条边界的含义是：

- 本地工具默认不再通过“额外一层 MCP 包装”再回调自己
- `MCP` 主要保留给 `Zotero` 这类真实外接系统

和当前 research agent 行为直接相关的补充是：

- `ResearchSupervisorAgent` 在正常运行时会通过 LLM 做 structured decision
- 如果某些单测或本地 stub 没有注入 LLM adapter，Supervisor 会退化成保守行为
- 这种退化路径只用于保证可测性，不代表线上 research orchestration 的真实能力边界

后端启动时会在 `lifespan` 里主动初始化底层依赖，当前重点是：

- `Milvus.ensure_schema()`
- `Neo4j.connect()`（仅在启用 `GRAPH_STORE_PROVIDER=neo4j` 时）

这意味着：

- `Milvus` 没起来，后端不会完成启动
- `Neo4j` 没起来，后端也不会完成启动

后端日志如果停在：

```text
Waiting for application startup.
```

但一直没有：

```text
Application startup complete.
```

通常就是底层依赖没有准备好。

## 5. 配置分层

### 5.1 最小本地可运行配置

如果你现在的目标只是：

- 页面能打开
- 前后端能联通
- 不先折腾 `Milvus / Neo4j / MySQL`

建议先用最小退化配置。

示例：

```env

LOG_LEVEL=INFO

API_KEY_ENABLED=false

```

这套配置的意义：

- 后端能启动
- 前端能正常打开研究工作区
- 不依赖外部模型服务
- 不依赖 `Milvus`
- 不依赖 `Neo4j`

限制：

- 回答质量会明显弱于完整业务配置
- 图谱检索和真实 embedding 不可用
- 高质量 research 工作流效果有限

但这套最小配置仍然足够验证大部分主线是否“结构正确”：

- Supervisor 调度是否正常
- research task / conversation / workspace 持久化是否正常
- 候选论文搜索与导入编排是否正常
- collection QA 路由是否正常
- `PaperAnalysisAgent` 的 evidence-aware analysis 链路是否正常

### 7.3 启动 MySQL / MariaDB

```env

MYSQL_USER=root

```

并且没有显式把：


改掉，那么真实 API 路径里的 session memory 会尝试使用 MySQL。

在 Ubuntu / WSL 上，常见启动方式是：

```bash
sudo systemctl start mysql
```

如果你的系统实际注册的是 `MariaDB` 服务名，则改用：

```bash

```

```bash
mysql -h 127.0.0.1 -P 3306 -u root -p
```

第一次初始化时，至少要保证业务库存在：

```sql
CREATE DATABASE IF NOT EXISTS ecs CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
SHOW DATABASES LIKE 'ecs';
```

当前项目里的 `session_memory` 表通常不需要手工创建。代码在第一次真正使用 session memory 时会自动执行 schema 初始化。

```env

```

这样会回退到内存会话记忆，不改变主编排，只是关闭 MySQL 这层持久化。

### 7.4 WSL 下桥接 Windows Zotero

如果你的实际运行方式是：

- `Research-Copilot` 后端运行在 `WSL/Linux`
- `Zotero` 桌面端运行在 `Windows`

那么不能直接假设下面这条地址在 WSL 内也可用：

```text
http://127.0.0.1:23119
```

原因：

- Windows 里的 `127.0.0.1` 是 Windows 自己的 localhost
- WSL 里的 `127.0.0.1` 是 WSL 自己的 localhost
- `Zotero Connector` 默认监听的是 Windows 本地 `23119`

这类场景推荐使用下面这条桥接链路：

```text
WSL 127.0.0.1:23119
-> socat
-> Windows WSL vEthernet 地址:23119
-> Windows portproxy
-> Windows 127.0.0.1:23119
-> Zotero
```

#### 7.4.1 先确认 Windows 上的 Zotero 正常

在 Windows PowerShell 中执行：

```powershell
curl http://127.0.0.1:23119/connector/ping
```

如果返回 `200 OK` 且正文包含 `Zotero is running`，说明 Windows 上的 Zotero 本体正常。

#### 7.4.2 在 Windows 上创建端口代理

以管理员身份打开 Windows PowerShell，执行：

```powershell
netsh interface portproxy add v4tov4 listenaddress=0.0.0.0 listenport=23119 connectaddress=127.0.0.1 connectport=23119
netsh advfirewall firewall add rule name="Zotero-23119" dir=in action=allow protocol=TCP localport=23119
```

查看规则是否存在：

```powershell
netsh interface portproxy show all
```

#### 7.4.3 找到 Windows 在 WSL 虚拟网卡上的地址

在 Windows PowerShell 里运行：

```powershell
ipconfig
```

通常应查看：

```text
vEthernet (WSL ...)
```

对应的 IPv4 地址。

例如：

```text
172.23.160.1
```

注意：

- 优先使用 `vEthernet (WSL ...)` 的地址
- 不要优先使用外网口或校园网口地址

#### 7.4.4 直接从 WSL 访问 Windows 地址时可能返回 400

在 WSL 中直接请求：

```bash
curl --noproxy '*' -v http://172.23.160.1:23119/connector/ping
```

可能会出现：

```text
HTTP/1.1 400 Bad Request
server: httpd.js
```

这通常不是 Zotero 没开，而是 Zotero 的本地服务不接受 `Host: 172.23.160.1:23119` 这类请求头。

因此，推荐继续在 WSL 本地做一层转发，把项目访问地址保持为 `127.0.0.1`。

#### 7.4.5 在 WSL 中用 socat 建立本地桥接

如果还没安装：

```bash
sudo apt update
sudo apt install -y socat
```

启动桥接：

```bash
socat TCP-LISTEN:23119,bind=127.0.0.1,reuseaddr,fork TCP:172.23.160.1:23119
```

这里的 `172.23.160.1` 需要替换成你机器上 `vEthernet (WSL ...)` 的实际 IPv4 地址。

注意：

- 跑 `socat` 的终端不能关闭
- 更稳妥的方式是单独开一个终端专门跑这条命令

#### 7.4.6 `.env` 中 Zotero 地址保持为 WSL 本地地址

完成桥接后，项目配置应保持为：

```env
ZOTERO_LOCAL_ENABLED=true
ZOTERO_LOCAL_BASE_URL=http://127.0.0.1:23119
ZOTERO_LOCAL_USER_ID=0
ZOTERO_LOCAL_TIMEOUT_SECONDS=20
```

也就是说：

- 不要把 `.env` 长期改成 `http://172.x.x.x:23119`
- 让项目始终访问 WSL 本地 `127.0.0.1`
- `socat` 负责把请求转发到 Windows

#### 7.4.7 在 WSL 中验证桥接是否成功

先绕过代理测试：

```bash
curl --noproxy '*' -v http://127.0.0.1:23119/connector/ping
```

预期结果：

```text
HTTP/1.0 200 OK
<!DOCTYPE html><html><body>Zotero is running</body></html>
```

如果你在 WSL 中配置了代理，还需要注意下面这些环境变量可能会劫持本地请求：

- `http_proxy`
- `https_proxy`
- `all_proxy`
- `HTTP_PROXY`
- `HTTPS_PROXY`
- `ALL_PROXY`

启动后端前建议执行：

```bash
unset http_proxy https_proxy HTTP_PROXY HTTPS_PROXY
unset all_proxy ALL_PROXY
export no_proxy=localhost,127.0.0.1,::1
export NO_PROXY=localhost,127.0.0.1,::1
```

### 7.5 启动后端

确保下面这些依赖已经准备好：

- `Milvus` 在 `127.0.0.1:19530`
- `Neo4j` 在 `127.0.0.1:7687`
- 如果 `.env` 中启用了 MySQL session memory：`MySQL` 在 `127.0.0.1:3306`
- Zotero Bridge 在 `127.0.0.1:23119`（可选）

## 9. 常见报错与排查

### 9.1 前端报 `connect ECONNREFUSED 127.0.0.1:8000`

原因：

- 后端没有启动
- 或后端启动后在 `lifespan` 阶段失败退出

先检查：

```bash
curl -i -s http://127.0.0.1:8000/health
```

### 9.2 后端卡在 `Waiting for application startup.`

原因通常是：

- `Milvus` 没准备好
- `Neo4j` 没准备好

当前代码会在启动时主动执行：

- `Milvus.ensure_schema()`
- `Neo4j.connect()`

所以这不是“接口第一次调用才报错”，而是“应用启动阶段直接失败”。

### 9.3 `unknown shorthand flag: 'f' in -f`

这是 `docker compose` 不可用，不是 compose 文件语法错误。

改用：

```bash
docker-compose -f docker-compose.milvus.yml up -d
```

### 9.4 Neo4j 报 `Couldn't connect to localhost:7687`

说明：

- 本地 Neo4j 没启动
- 或 Bolt 端口不是 `7687`
- 或 `.env` 中 `NEO4J_URI` 不匹配

当前项目默认期望：

```env
NEO4J_URI=bolt://localhost:7687
```

### 9.5 Neo4j 报 `JAVA_HOME is not defined correctly`

说明：

- 本地 Neo4j 没找到 Java

如果使用 conda 环境里的 Java：

```bash
source /home/myc/miniconda3/etc/profile.d/conda.sh
conda activate Research-Copilot
export JAVA_HOME="$CONDA_PREFIX"
export PATH="$JAVA_HOME/bin:$PATH"
```

前提是这个 conda 环境里已经安装了 `openjdk`

### 9.6 MySQL 报连接失败或 `Access denied`

常见原因：

- `mysql` / `mariadb` 服务没有启动
- `.env` 里的 `MYSQL_USER`、`MYSQL_PASSWORD`、`MYSQL_DATABASE` 不匹配
- 你本来不想用 MySQL，但忘了把 `SESSION_MEMORY_PROVIDER` 改成 `memory`

先检查服务和连通性：

```bash
mysql -h 127.0.0.1 -P 3306 -u root -p
```

再检查业务库是否存在：

```sql
SHOW DATABASES LIKE 'ecs';
```

如果你只是临时联调，不想依赖 MySQL，直接把 `.env` 改成：

```env
SESSION_MEMORY_PROVIDER=memory
```

### 9.7 Milvus 相关错误

如果后端日志里出现 Milvus 无法连接，先检查：

```bash
cd /home/myc/Research-Copilot
docker-compose -f docker-compose.milvus.yml ps
docker-compose -f docker-compose.milvus.yml logs -f milvus
```

常见问题：

1. **healthz 返回 `component proxy state is Abnormal`**
   说明 Milvus 内部组件没有完全启动。可能原因：
   - etcd 没启动成功（检查 `docker-compose -f docker-compose.milvus.yml logs etcd`）
   - Docker 代理设置干扰了容器间通信（见 9.11）
   - 数据目录损坏（尝试 `docker-compose down && sudo rm -rf .data/milvus/etcd/* .data/milvus/data/* && docker-compose -f docker-compose.milvus.yml up -d`）

2. **Milvus 容器反复崩溃（exit 134 / SIGABRT）**
   这是 WSL2 + Milvus 的已知问题。确保：
   - 使用 `docker-compose` 方式启动（compose 文件已包含 `security_opt: seccomp:unconfined`）
   - compose 文件中所有服务已设置 `no_proxy="*"` 阻止代理干扰

3. **embedding 模型维度和现有 collection 不一致**
   会看到维度不匹配错误。这时要么：
   - 调整 `MILVUS_DIMENSION`
   - 要么重建 collection

仓库里已经提供了重置脚本：

```bash
cd <repo-root>
/home/myc/miniconda3/envs/Research-Copilot/bin/python scripts/reset_milvus_collection.py
```

### 9.8 Zotero local 在 Windows 能通、WSL 里不通

典型现象：

- Windows PowerShell 里：

```powershell
curl http://127.0.0.1:23119/connector/ping
```

返回 `200 OK`

- 但 WSL 里：

```bash
curl http://127.0.0.1:23119/connector/ping
```

返回连接失败

这说明：

- Zotero 本体是好的
- 问题是 `WSL localhost` 和 `Windows localhost` 不是同一个地址

按 `7.4 WSL 下桥接 Windows Zotero` 一节处理即可。

### 9.9 WSL 请求 Zotero 时被代理拦截

典型现象：

```text
Uses proxy env variable http_proxy == ...
Uses proxy env variable all_proxy == ...
HTTP/1.1 502 Bad Gateway
```

这说明请求先被发到了你的本地代理，而不是直接访问 Zotero。

先绕过代理验证：

```bash

```

必要时清掉代理变量：

```bash
unset http_proxy https_proxy HTTP_PROXY HTTPS_PROXY
unset all_proxy ALL_PROXY
```

### 9.11 Milvus 内部组件连接失败（Docker 代理干扰）

典型现象：

- Milvus 容器启动后立即崩溃，exit code 134（SIGABRT）
- 日志中出现 `panic: failed to create etcd client: context deadline exceeded`
- 或 healthz 返回 `component proxy state is Abnormal` 且长时间不恢复

根因：

如果 Docker daemon 配置了 HTTP 代理（例如通过 `systemd` 的 `/etc/systemd/system/docker.service.d/proxy.conf`），**所有容器会继承代理环境变量**。这导致 Milvus 容器内部访问 `etcd:2379` 和 `minio:9000` 的请求被转发到代理服务器，代理无法解析 Docker 内部 DNS 名称，返回 502 Bad Gateway。

解决方法：

当前 `docker-compose.milvus.yml` 已经为所有服务显式设置了代理无关变量：

```yaml
environment:
  http_proxy: ""
  https_proxy: ""
  HTTP_PROXY: ""
  HTTPS_PROXY: ""
  no_proxy: "*"
  NO_PROXY: "*"
```

如果你之前手动执行过 `docker run` 创建的 Milvus 容器（而非通过 compose），需要先删除旧容器后重新用 compose 启动：

```bash
docker rm -f research-copilot-milvus research-copilot-etcd research-copilot-minio 2>/dev/null
cd /home/myc/Research-Copilot
docker-compose -f docker-compose.milvus.yml up -d
```

如果数据目录损坏（etcd 反复 panic），可以安全清理后重启：

```bash
cd /home/myc/Research-Copilot
docker-compose -f docker-compose.milvus.yml down
sudo rm -rf .data/milvus/etcd/* .data/milvus/data/*
docker-compose -f docker-compose.milvus.yml up -d
```

❗ 清理数据目录会丢失所有已索引的向量数据，导入过的论文需要重新 parse/index。
